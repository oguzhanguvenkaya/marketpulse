# MarketPulse - Kapsamlı Kod İnceleme & Mimari Raporu (Code Review)

Bu rapor, MarketPulse Pazaryeri Veri Analiz Platformu'nun mevcut backend (FastAPI, SQLAlchemy, Celery), frontend (React, Vite, TailwindCSS) ve genel sistem mimarisi üzerinden gerçekleştirilen mimari analizleri ve incelemeleri içermektedir.

---

## 1. 📊 Yönetici Özeti ve Genel Değerlendirme

MarketPulse projesi, genel mimari kararları, iş mantığı dökümleri ve modüler klasör hiyerarşisi ile övgüyü hak eden olgun bir yapıya sahiptir. Servisler arası izolasyon sağlanmış ve modern teknolojilerle (GPT-4o, Playwright, ScraperAPI vs.) entegre edilmiştir. Ancak, uygulamanın büyüme aşamasında özellikle **veritabanı sorgu optimizasyonları (Performance - N+1 sorgular)** ve **bağlantı yönetimi (Connection Handling)** konularında ciddi darboğazlar yaşatabilecek teknik borçları mevcuttur.

**👍 İyi Tarafları (Pros):**
*   **Modüler Dosya Yapısı:** `backend/app/api`, `backend/app/services`, `backend/app/db` şeklindeki katmanmanlı yapı (Domain-Driven Design esintileri) projenin bakımını oldukça kolaylaştırıyor.
*   **Config Yönetimi:** Gizli bilgileri kod içerisine gömmek (hardcode) yerine `pydantic-settings` (bkz: `config.py`) üzerinden dinamik environment variables kullanılmış. Harici proxy API'leri fallback mantığı ile tasarlanmış.
*   **Database Retry Pattern:** `app/api/routes.py` içindeki `_run_read_query_with_retry` metodu `OperationalError` durumunda otomatik kurtarma sağlayarak veritabanı kararlılığını artırıyor.
*   **Frontend Servis İzolasyonu:** API istekleri tek bir `api.ts` servisinde toplanarak axios middleware'i (Interceptor) temiz bir şekilde konumlandırılmış.

**👎 Kötü Tarafları (Cons):**
*   **N+1 Sorgu Problemleri:** `routes.py` içindeki sorgularda lazy loading kaynaklı ciddi N+1 hataları mevcut. (Özellikle liste döndüren endpoint'lerde).
*   **Redis Sağlık Kontrolü Maliyeti:** `_queue_reachable()` metodunda her ping işleminde yeni bir Redis bağlantısı (Connection) oluşturulup kapatılıyor, bu durum yüksek trafik altında "Connection Pool Exhaustion" sorunlarına yol açabilir.
*   **Kendi Cache Mantığını Geliştirmek:** Frontend tarafında standart endüstri çözümleri (Örn. `@tanstack/react-query`) kullanmak yerine `queryCache.ts` adında custom bir önbellek yazılması esnekliği ve standartları zedeliyor.
*   **Güvenlik (Authorization):** API kimlik doğrulamasında JWT (Token) veya RBAC tabanlı yetkilendirme yerine sadece tek bir `X-API-Key` (İç API Key) kontrolüne dayanılmış. Bu durum canlı ortamlarda (multi-tenant) veri hırsızlığı riskini barındırıyor.

## 2. 🏗️ Mimari, Modülerlik ve Kod Kalitesi

*   **Modülerlik:** MVC'ye benzetilebilecek Servis-View-Controller (Router) yapısına sadık kalınmış. Sorumlulukların ayrılması ilkesi (Separation of Concerns) `services` klasörü altına iş mantıklarının taşınması ile güzel yakalanmış.
*   **Clean Code & DRY:** Genel yapı DRY prensibine uygun, fakat `api/routes.py` dosyasının **2000 satırı aşması** bir "God Object" anti-pattern alarmı veriyor! Modüllere (`products_router.py`, `search_router.py` vb.) daha fazla parçalanmalı.
*   **ORM İlişkileri:** `models.py` içerisinde tanımlanan veritabanı bağıntılarında (`relationship()`) `lazy="selectin"` veya `lazy="joined"` gibi modern eager loading parametreleri tanımlanmadığı için varsayılan load stratejisi bırakılmış.

---

## 3. 🔒 Güvenlik Analizi (Security Audit)

*   **Hassas Veriler:** Şifreler ve ortam değişkenleri `.env` üzerinden yönetilse de `.env.example` veya repoya yanlışlıkla eklenebilecek `SCRAPER_API_KEY` vb. secret'lar açısından `.gitignore` kullanımının katı olduğuna dikkat edilmelidir. Kaynak koda hardcoded bir anahtar tespit edilmedi (Olumlu).
*   **Yetkilendirme:** `app/core/security.py` dosyasındaki `require_mutating_api_key` bir Middleware gibi kullanılmış. İlgili `X-API-Key` eşleşmesi ile herkese açık olmayan işlemlere koruma gelmiş ancak statik tekil anahtarlar güvenlik sızıntısında tüm sistemi kompromize eder. Üretim ortamında her müşteri/servis için farklı bir rol/anahtar veya JWT OAuth2 mimarisine geçiş önerilir.
*   **CORS:** `CORS_ALLOWED_ORIGINS` değişkenine virgülle bağlanmış origin desteği esnek ve doğru ayarlanmış. Localhost testleri için spesifik port bırakılmış, wildcard (`*`) kullanılmadığı için güvenli bulunmuştur.

## 4. 🎨 UI/UX ve Frontend İyileştirmeleri

*   **Kullanıcı Arayüzü & Mimarisi:** TypeScript ve React kullanılarak bileşen (component) tabanlı bir yaklaşım izlenmiş. Ancak state yönetimi ve cache iptali için `api.ts` içerisine `invalidateDashboardCache` gibi business mantıklar manuel implement edilmiş.
*   **UX Tavsiyesi:** UI üzerinde tekrar eden API çağrıları manuel timeout veya "queryCache.ts" ile yönetiliyor. Bunun yerine TanStack Query entegrasyonu ile `loading`, `isError`, ve `isFetching` bildirimleri UI'a daha soft yansıtılarak Skeleton yüklemeleri (Skeleton Loaders) eklenmelidir. Frontend uygulamasında veri çekerken "Silent Error" durumunda bile kullanıcıya nazik "Daha sonra tekrar deneyin" toast mesajları (Örn: `react-toastify`) verilmesi kilit öneme sahiptir.

## 5. ⚡ Performans ve Kaynak Yönetimi

Altyapıyı krize sokabilecek veritabanı (Performans) hataları mevcuttur:

*   **N+1 Problemi (Darboğaz):** 
    `app/api/routes.py` satır ~615'teki `list_products` metodunda:
    ```python
    products = query.order_by(desc(Product.created_at)).limit(limit).all()
    for p in products:
        latest = db.query(ProductSnapshot).filter(ProductSnapshot.product_id == p.id).first()
    ```
    Burada 50 ürün listelendiğinde 1 adet ürün sorgusu ve üstüne ürünler döndüğü kadar (50 adet) `ProductSnapshot` veritabanı sorgusu atılmaktadır (Toplam 51 Sorgu). Bu, sistemi veritabanı I/O maliyetine boğar!
    
*   **Senkron Redis Ping:**
    `app/main.py` dosyasındaki `_queue_reachable()` fonksiyonunda:
    ```python
    client = Redis.from_url(settings.REDIS_URL, ...)
    bool(client.ping())
    client.close()
    ```
    Healthcheck `/health` rotasında her istekte TCP Redis Socket'i açıp kapatmak, yüksek throughput'ta port bitmesine (EADDRINUSE) ve bekleme/kapanma (TIME_WAIT) birikimlerine yol açar. Bir `ConnectionPool` tanımlanmalı ve tekrar kullanılmalıdır.

## 6. 📝 Test, Hata Yönetimi ve Dokümantasyon

*   **Hata Yönetimi (Error Handling):** FastApi'nin global Exception Handler'ı (Global Error Middleware) tam olarak override edilmemiş. Servis katmanında `except Exception as e: logger.error(f"Search task {task_id} failed: {e}")` sonrasında işlemin "sessiz başarısızlık" modunda `status = "failed"` yapılması ve kullanıcıya bir Webhook veya standart 400/500 JSON Payload verilmemesi izlenebilirliği (Observability) zayıflatır.
*   **Dokümantasyon:** Proje `.md` dosyaları (README, ARCHITECTURE) bakımından inanılmaz zengin ve yeterlidir. Geliştiricilerin onboarding sürecini sıfıra indiren bir yapıya sahip.

## 7. 🚀 Aksiyon Planı ve Refactoring Önerileri (EN ÖNEMLİ KISIM)

Projeyi teknik borçlardan arındırmak için uygulanması gereken en kritik ilk 3 adım:

1.  **ORM İlişkilerini Eager Loading (Joined) olarak revize etmek.** (N+1 engellemesi)
2.  **Global Connection Pool (Redis) mekanizmasına geçmek.** (Network/Socket patlamasını önleme)
3.  `routes.py`'nin (2145 satır) alan adlarına (Domain) göre (Price, URL, Product vs.) küçük router modüllerine ayrıştırılması.

### Mimaride Yapılması Gereken Kod Refactoring Örnekleri:

#### Refactoring 1: N+1 Sorgu Optimizasyonu (Mevcut Kod: app/api/routes.py)

**Mevcut Kod (Before): N+1 Darboğazı**
```python
@router.get("/products", response_model=List[ProductResponse])
async def list_products(limit: int = 50, db: Session = Depends(get_db)):
    products = db.query(Product).order_by(desc(Product.created_at)).limit(limit).all()
    result = []
    # KÖTÜ: Her bir ürün için for döngüsünde DB'ye gidip tekrar sorgu atılıyor! (1 + 50 Sorgu)
    for p in products:
        latest = db.query(ProductSnapshot)\
                   .filter(ProductSnapshot.product_id == p.id)\
                   .order_by(desc(ProductSnapshot.snapshot_date)).first()
        # append işlemleri ...
```

**Olması Gereken Kod (After): Eager Loading & Outer Join Kullanımı**
```python
from sqlalchemy.orm import selectinload

@router.get("/products", response_model=List[ProductResponse])
async def list_products(limit: int = 50, db: Session = Depends(get_db)):
    # İYİ: `selectinload` ile veritabanında tüm Snapshot'ları 2 verimli sorguyla alır.
    products = (db.query(Product)
                  .options(selectinload(Product.snapshots))
                  .order_by(desc(Product.created_at))
                  .limit(limit)
                  .all())
    
    result = []
    for p in products:
        # DB'ye bir daha sorulmaz, hafızaya alınan snapshots listesi Python'da eleman olarak sıralanır:
        latest = None
        if p.snapshots:
            latest = sorted(p.snapshots, key=lambda x: x.snapshot_date, reverse=True)[0]
        # append işlemleri ...
```

#### Refactoring 2: Redis Connection Yorgunluğunu Önleme (Mevcut Kod: app/main.py)

**Mevcut Kod (Before): Health Check**
```python
def _queue_reachable() -> bool:
    client = None
    try:
        # KÖTÜ: İstek başına yeni instance socket TCP açma işlemi ve Timeout riski.
        client = Redis.from_url(settings.REDIS_URL, socket_connect_timeout=1, socket_timeout=1)
        return bool(client.ping())
    except Exception:
        return False
    finally:
        if client is not None:
            client.close()
```

**Olması Gereken Kod (After): Connection Pooling**
```python
from redis import ConnectionPool, Redis

# Redis URL pool'u globalde ve uygulama ilk açıldığında tekil olarak init edilir
redis_pool = ConnectionPool.from_url(
    settings.REDIS_URL,
    socket_connect_timeout=1,
    socket_timeout=1,
    max_connections=10  # Maksimum limit belirtilir
)

def _queue_reachable() -> bool:
    try:
        # İYİ: Mevcut açık TCP soketlerinden (pool) birini kullanıp hemen bırakır.
        with Redis(connection_pool=redis_pool) as client:
            return bool(client.ping())
    except Exception as e:
        logger.error(f"Redis healthcheck failed: {e}")
        return False
```
